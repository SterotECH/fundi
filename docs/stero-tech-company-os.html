User Stories

Written as "As Stero, I want to… so that…" for every major feature. Grouped by sprint. These stories define acceptance — if the feature doesn't satisfy the story, the story isn't done.

Sprint 1 The Usable Core Weeks 1–3
Authentication
As Stero, I want to log in with my email and password so that my business data is protected behind a credential wall.
As Stero, I want my session to stay alive via token refresh so that I don't have to re-login every hour while working.
As Stero, I want to log out cleanly so that I can end my session from any device.
Client Management
As Stero, I want to create a client profile with name, contact person, phone, email, address, and school type so that I have a single source of truth for every institution I serve.
As Stero, I want to view a searchable, paginated list of clients so that I can quickly find any school.
As Stero, I want to view a client detail page showing all linked proposals, projects, invoices, and payments so that I can understand the full relationship at a glance.
As Stero, I want to archive (not delete) clients so that I preserve historical data while keeping my active list clean.
Lead Management
As Stero, I want to add a lead with name, contact info, source, and notes so that I can track prospects before they become paying clients.
As Stero, I want to convert a lead to a client with one action so that I don't re-enter data when a prospect commits.
As Stero, I want to mark a lead as dead or inactive so that my pipeline only shows live opportunities.
Proposal Tracker
As Stero, I want to create a proposal linked to a client with title, description, amount, and deadline so that I have a formal record of every pitch.
As Stero, I want to move a proposal through status stages (draft → sent → negotiating → won → lost) so that I can track where every deal stands.
As Stero, I want to see a visual deadline indicator when a proposal is approaching or has passed so that I never miss a follow-up window.
As Stero, I want to convert a won proposal into a project with one click so that I don't re-enter the same data twice.
Dashboard
As Stero, I want to see upcoming proposal deadlines on my dashboard so that I can plan my follow-up calls for the week.
As Stero, I want to see counts of open proposals by status so that I know the health of my pipeline at a glance.
As Stero, I want to see my currently active projects so that I know what I should be building right now.
Sprint 2 The Money Layer Weeks 4–6
Invoice Management
As Stero, I want to create an invoice linked to a project and client with auto-generated invoice numbers so that every invoice is uniquely identifiable.
As Stero, I want to add line items to an invoice so that I can document exactly what I'm charging for.
As Stero, I want to track invoice status (draft, sent, partial, paid, overdue) so that I know the collection state of every invoice.
As Stero, I want to see which invoices are overdue so that I can follow up proactively.
Payment Tracking
As Stero, I want to record a payment specifying amount in GHS, method (MTN MoMo, Vodafone Cash, AirtelTigo, bank transfer), reference number, and date so that I have a complete payment audit trail.
As Stero, I want the invoice status to automatically update to "partial" or "paid" based on total payments recorded so that I don't manage it manually.
As Stero, I want to see total revenue collected vs invoiced per client so that I know who still owes me money.
Time Logging
As Stero, I want to log hours worked on a project with a date, duration, and description so that I can track how much time I spend per client.
As Stero, I want to mark time logs as billable or non-billable so that I can distinguish client time from internal work.
As Stero, I want to see total hours logged per project so that I know if I'm over-budget on time.
Deadline Reminders (Celery)
As Stero, I want to receive an in-app notification when a proposal deadline is 3 days away so that I never let a pitch expire without action.
As Stero, I want to receive a notification when an invoice becomes overdue so that I can chase payment immediately.
As Stero, I want Celery to run these checks on a daily schedule so that reminders are fully automatic.
Sprint 3 The Intelligence Layer Weeks 7–9
AI Business Assistant
As Stero, I want to ask questions like "How much revenue did I collect this month?" so that I get instant answers without building reports myself.
As Stero, I want the AI to read my live CRM data when answering so that responses are accurate and current.
As Stero, I want to ask the assistant to draft a follow-up email for a specific proposal so that I can copy-paste a professional message without writing it from scratch.
As Stero, I want the assistant to summarise an overdue client's outstanding balance in one message so that I'm prepared before making a payment call.
Pipeline Analytics
As Stero, I want to see total revenue collected broken down by month so that I can spot seasonal trends.
As Stero, I want to see my proposal win rate so that I can evaluate how good my pitching is.
As Stero, I want to see revenue and time invested per client so that I can identify my most and least profitable relationships.
As Stero, I want to see average time-to-close per proposal so that I can benchmark how long deals typically take.
Polish & Mobile
As Stero, I want all screens to work comfortably on my phone so that I can check the CRM between meetings without a laptop.
As Stero, I want consistent loading states, empty states, and error states throughout the app so that it feels professional when I demo it to a client.

API Contract

All endpoints are prefixed with /api/v1/. Authentication is required on all endpoints except /auth/login/ and /auth/refresh/. Standard responses: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 404 Not Found.

State machine enforcement

Proposal and Invoice status changes must go through dedicated /transition/ and /send/ endpoints — never via PATCH directly. The backend enforces allowed transitions and rejects invalid ones with 400.

Auth
POST/auth/login/Obtain access + refresh tokens. Body: {email, password}. Returns: {access, refresh, user}.
POST/auth/refresh/Refresh access token. Body: {refresh}. Returns: {access}.
POST/auth/logout/Blacklist refresh token. Body: {refresh}. Returns: 204.
GET/auth/me/Get current user profile.
PATCH/auth/me/Update profile or password. Body: {full_name?, password?}.
Clients
GET/clients/List clients. Filterable by is_archived, searchable by name. Paginated.
POST/clients/Create client. Body: {name, type, contact_person, email, phone, address, region, notes}.
GET/clients/{id}/Client detail with nested summary counts.
PATCH/clients/{id}/Update client fields (partial update supported).
DELETE/clients/{id}/Soft-archive client. Sets is_archived=True. Returns 204.
GET/clients/{id}/proposals/All proposals for this client.
GET/clients/{id}/invoices/All invoices for this client.
GET/clients/{id}/projects/All projects for this client.
Leads
GET/leads/List leads. Filterable by status. Paginated.
POST/leads/Create lead. Body: {name, contact_person, email, phone, source, notes}.
GET/leads/{id}/Lead detail.
PATCH/leads/{id}/Update lead fields.
POST/leads/{id}/convert/Convert lead to client. Populates converted_to_client. Returns {client}.
Proposals
GET/proposals/List proposals. Filterable by status, client_id. Orderable by deadline.
POST/proposals/Create proposal. Body: {client_id, title, description, amount, deadline, notes}.
GET/proposals/{id}/Proposal detail.
PATCH/proposals/{id}/Update proposal fields (draft/sent only for amount changes).
POST/proposals/{id}/transition/Change status via state machine. Body: {status: "sent"|"negotiating"|"won"|"lost"}.
POST/proposals/{id}/convert/Convert won proposal to project. Body: {title?, start_date, due_date}. Returns {project}.
DELETE/proposals/{id}/Hard delete. Draft status only. Returns 204.
Projects
GET/projects/List projects. Filterable by status, client_id.
POST/projects/Create project manually. Body: {client_id, title, description, start_date, due_date, budget, proposal_id?}.
GET/projects/{id}/Project detail with milestones and time summary.
PATCH/projects/{id}/Update project fields.
GET/projects/{id}/milestones/List milestones for project.
POST/projects/{id}/milestones/Add milestone. Body: {title, description, due_date, order}.
PATCH/projects/{id}/milestones/{mid}/Update or complete a milestone. Body: {completed: true}.
DELETE/projects/{id}/milestones/{mid}/Delete milestone. Returns 204.
GET/projects/{id}/timelogs/Time logs for project with total hours summary.
GET/projects/{id}/invoices/Invoices linked to project.
Invoices
GET/invoices/List invoices. Filterable by status, client_id.
POST/invoices/Create invoice with line items. Body: {client_id, project_id?, due_date, line_items: [{description, quantity, unit_price_ghs}]}.
GET/invoices/{id}/Invoice detail with line items and payment history.
PATCH/invoices/{id}/Update invoice. Draft status only.
POST/invoices/{id}/send/Mark invoice as sent. Updates status and sets sent_date.
DELETE/invoices/{id}/Delete draft invoice only. Returns 204.
GET/invoices/overdue/All overdue invoices across all clients.

UI/UX Guidelines & Wireframes

Design language

Functional minimalism with bento grid structure. Clean flat surfaces, 0.5px borders, generous whitespace. The dashboard is built from a 12-column bento grid where tile size communicates importance. Larger tiles carry more complex content; small tiles hold single numbers.

Color semantics

Colors are not decorative — they encode meaning. Every badge, border, and background must follow this system consistently across every screen.

Interaction patterns

  • Every list row is clickable and navigates to a detail view. No separate "View" buttons in tables.
  • Status changes happen via a contextual action menu (three-dot icon on hover), never by navigating away.
  • Forms open as right-side drawers sliding in from the right — context remains visible while filling in fields.
  • Destructive actions (archive, delete) use inline confirmation ("Are you sure? Confirm / Cancel"), not a blocking modal dialog.
  • Every empty state includes a primary CTA — never show an empty list without "Create your first X".
  • Mobile is tested on every tile before a sprint closes. Ghana is a mobile-first market.

Critical Design Decisions

These are the decisions that are painful to change after coding begins. Each one is flagged here so you make it consciously before writing a single migration.

Decision 01
UUIDs everywhere, not integer IDs

Use uuid.uuid4 as the default PK on every model. This is a public GitHub repo — sequential integer IDs leak your record counts to anyone who inspects an API response ("id": 47 tells them you have 47 clients). UUIDs are opaque. The performance cost on Postgres at your data volumes is negligible. Set this once and never revisit it.

Decision 02
Organisation model from day one, not retrofitted

Add the Organisation model before any other model. Add organisation (FK → Organisation) to every model you build. In all your views, filter by request.user.organisation. This is one extra line per query in Sprint 1 and it's what makes the entire system multi-tenant in Sprint 4. Retrofitting this after 20 models and 50 views exist is a project of its own.

Decision 03
Custom User model before the first migration

Set AUTH_USER_MODEL = 'accounts.User' in settings before you run manage.py migrate for the first time. Django's built-in user model swap is catastrophic to do after migrations exist — it requires wiping and recreating your entire database. This is a five-minute step that saves a three-hour disaster.

Decision 04
JWT token storage — HttpOnly cookie, not localStorage

Do not store the access token in localStorage — it is vulnerable to XSS attacks. Store the access token in memory (React state or a module-level variable) and the refresh token in an HttpOnly cookie set by Django. Configure SimpleJWT to set the refresh cookie server-side. This is worth getting right in Sprint 1 because retrofitting it disrupts every authenticated flow.

Decision 05
Decimal, never Float, for all monetary values

Use DecimalField(max_digits=12, decimal_places=2) for every GHS amount in Django models. Use Python's Decimal type (not float literals) for any arithmetic. float arithmetic produces rounding errors — 0.1 + 0.2 returns 0.30000000000000004 in Python. On financial data this is unacceptable and will silently corrupt your totals.

Decision 06
Soft delete everywhere except draft records

Clients get an is_archived flag — never hard deleted, because hard-deleting a client cascades to proposals, invoices, and payments and destroys your financial history. Draft proposals and draft invoices can be hard-deleted (they haven't been sent, so no external record exists). Paid invoices must never be deletable — enforce this in the view layer, not only in the frontend.

Decision 07
Invoice number generation via Postgres sequence

Auto-generating STERO-2025-001 requires an atomic counter. Do not use MAX(invoice_number) + 1 — it has a race condition if you ever run multiple Celery workers or gunicorn processes simultaneously. Use a Postgres sequence (CREATE SEQUENCE invoice_seq) accessed via a raw query in the model's save(). It is atomic by design.

Decision 08
Proposal state machine in a service function

The allowed transitions are: draft→sent, sent→negotiating, sent→won, sent→lost, negotiating→won, negotiating→lost. Enforce these in a transition_proposal(proposal, new_status) service function — not directly in the view. A plain dictionary mapping of {current_status: [allowed_next_statuses]} is sufficient. When a status goes to won or lost, automatically set decision_date = today.

Decision 09
AuditLog wired from Sprint 1, UI built later

Wire a Django post-save and post-delete signal that writes to AuditLog on every model create, update, and delete. Store the before/after diff as JSON. Build no UI for it in Sprint 1. When something goes wrong in production six months from now — a payment accidentally deleted, a proposal status changed incorrectly — you will have a complete audit trail to diagnose it. The cost is minimal; the value when needed is enormous.

Decision 10
AI context injection — plan the boundaries now

When the AI assistant calls the Anthropic API in Sprint 3, a build_ai_context() function will inject a snapshot of CRM data as a system prompt. Plan now that this function returns a capped summary — last 3 months of activity, top 10 clients by revenue, open proposals, overdue invoices. If you let it grow unboundedly, you will hit token limits and the assistant will silently truncate context. The frontend never sends raw CRM data to Anthropic — all data access is mediated through your Django backend.

Decision 11
Celery + Beat as Docker services from start of Sprint 2

Add the Celery worker and Celery Beat scheduler as Docker Compose services at the very beginning of Sprint 2 — not as an afterthought when you need your first scheduled task. A missing or misconfigured docker-compose.yml entry will cost an hour of environment debugging when you're trying to test deadline reminders. Wire the services first, write the tasks second.
