The Redis check matters more than it looks

Run docker compose exec django python -c "import redis; r = redis.from_url('redis://redis:6379'); r.ping()" and confirm it returns True. Celery Beat silently fails if the broker is unreachable, and you won't discover it until you're wondering why no notifications are firing in Week 5.

Build order rule

Always go backend-first within each week. A working endpoint with no UI is a testable, demonstrable thing. A UI that calls a non-existent endpoint is dead weight. Every screen in Week 5 should be wiring a React Query hook to an endpoint that already passes its pytest tests.

Migration strategy

Create one migration for Invoice + InvoiceLineItem together — they're created and destroyed as a unit and there's no benefit to separating them. Payment gets its own migration since it needs the Invoice table to exist first. TimeLog and Notification get individual migrations. Total: 4 new migration files in Sprint 2.

API contract — Sprint 2 endpoints

All endpoints prefixed /api/v1/. All require JWT auth. Standard responses: 200, 201, 204, 400, 401, 404. Decimal amounts always serialized as strings to preserve precision across JSON.

Invoices
GET/invoices/List invoices. Filter by status, client_id, project_id. Paginated.
POST/invoices/Create invoice with nested line items. Body: {client_id, project_id?, due_date, notes?, line_items: [{description, quantity, unit_price}]}. Auto-calculates totals.
GET/invoices/{id}/Invoice detail with nested line items and payment history. Includes amount_paid and amount_remaining computed fields.
PATCH/invoices/{id}/Update invoice. Only allowed when status is draft. Rejects updates to sent/paid invoices.

Never recalculate status asynchronously

When a payment is created or deleted, recalculate_invoice_status() runs synchronously within the same request cycle — before the response is sent. Do not offload this to a Celery task. If there's a delay between the API response and the status update, the UI will show a stale status and the user will be confused. The function is fast (one aggregate query) and belongs inline.

Invoice status logic

The recalculate_invoice_status(invoice) service function is the most critical piece of business logic in Sprint 2. It must be the single source of truth for every invoice status transition triggered by payment events. Call it from the payment post_save signal AND from the payment delete endpoint — nowhere else.

Partial + overdue edge case

A partially-paid invoice that passes its due date should become overdue — not stay partial. The Celery task handles this: it sweeps all invoices where due_date < today and status != paid, including those currently at partial. The recalculate_invoice_status() function also handles it: it always checks the due date before returning partial.

Invoice number generation

Run this in a data migration using migrations.RunSQL:

CREATE SEQUENCE IF NOT EXISTS stero_invoice_seq START 1;

In Invoice.save(), when invoice_number is empty and status transitions to sent:

SELECT nextval('stero_invoice_seq')
→ format as STERO-{year}-{number:04d}
→ e.g. STERO-2025-0007
Why Postgres sequence and not MAX + 1

MAX(invoice_number) + 1 is not atomic. Two simultaneous requests (you running Django dev server + Celery worker) can both read the same MAX value and generate duplicate invoice numbers. A Postgres sequence is atomic by design — nextval() always returns a unique value regardless of concurrency. This matters now and critically matters when you add Gunicorn workers later.

Provider reference validation rule

Do not validate the format of provider_reference. MTN MoMo references look like GH-MOMO-2025042900887, GCB bank transfer refs look like GCB-TXN-88234912, and Vodafone Cash refs have their own format. There is no cross-provider standard. Validate only that the field is non-empty when the method is not cash.

Effective rate calculation

The effective rate shown in the hours panel is project.budget / total_billable_hours. This is computed at query time in the /projects/{id}/timelogs/ endpoint — not stored on the model. When Sprint 3 adds analytics, this same calculation feeds the profitability-per-client chart without any schema changes.

Tasks must never call external services

Both Celery tasks only write to the database. They do not send email, call webhooks, or hit any external API. When you add email in Sprint 4, you add a separate task that reads from the Notification table — you don't touch these tasks. This keeps Sprint 2 tasks simple, testable, and rollback-safe.

Duplicate notification guard

Add a get_or_create check in both tasks keyed on (user, type, entity_id, created_at__date=today). This prevents a re-run or a Beat timing glitch from spamming the notification table with duplicate entries for the same invoice or proposal.

What Sprint 2 hands to Sprint 3

By the end of Sprint 2, every piece of financial data exists in your database in clean, queryable form. Sprint 3's analytics endpoints (/analytics/revenue/, /analytics/win-rate/, /analytics/clients/profitability/) are pure database aggregation queries against models that already exist. The intelligence layer's build_ai_context() function will read from Invoice, Payment, TimeLog, and Notification — all of which Sprint 2 populates with real data. Sprint 3 has zero new model migrations. It adds query endpoints, rule-based insights, and shared context-building on top of the foundation Sprint 2 built.
