The no-LLM decision

Sprint 3 delivers zero LLM calls. This is the right call for three concrete reasons.

Why no LLM in Sprint 3

You don't have enough data yet. An LLM reading 3 invoices and 5 time logs will produce vapid observations ("Your revenue this month is GHS 3,200"). The same data in a well-designed analytics screen tells you much more, instantly, at zero cost per query.

The context layer needs to be built first anyway. The future intelligence roadmap depends on a build_ai_context() function that serialises a structured snapshot of your CRM data. You build that function in Sprint 3 as shared infrastructure for Celery jobs, notifications, and later rule packs. When future sprints expand automation, the context is already tested and correct.

Rules beat LLMs for deterministic queries. "What's my win rate?" has one correct answer. A SQL aggregate cannot drift. The rule-based insights you build in Sprint 3 will be more reliable and faster than any probabilistic layer for structured financial questions, and they remain useful all the way through Sprint 8.

Analytics API contract

All endpoints prefixed /api/v1/analytics/. All require JWT auth. All are GET-only — pure read. All filter by request.user.organisation implicitly. No pagination — these are aggregate queries that return small summary objects, not lists.

Revenue
GET /analytics/revenue/ Monthly revenue from paid and partial payments for the last 12 months. Query param: months=12 (default). Returns {months: [{month: "2025-04", collected_ghs: "18400.00", invoiced_ghs: "22000.00"}], total_collected, total_outstanding}. Source: Payment.amount grouped by payment_date__year + payment_date__month.
GET /analytics/revenue/summary/ Flat summary for dashboard tiles: {this_month_collected, last_month_collected, mom_change_pct, ytd_collected, total_outstanding, overdue_count, overdue_total}. This replaces the hardcoded dashboard tiles.
Pipeline
GET /analytics/pipeline/ Proposal funnel + win metrics: {by_status: [{status, count, total_value_ghs}], win_rate_pct, avg_deal_value_ghs, avg_days_to_close, total_pipeline_value_ghs}. Only proposals with decision_date set count toward avg_days_to_close.
Clients
GET /analytics/clients/ Per-client profitability. Returns array sorted by revenue_ghs desc: [{client_id, client_name, invoiced_ghs, collected_ghs, outstanding_ghs, total_hours, billable_hours, effective_rate_ghs, open_proposals}]. Query param: sort_by=revenue|hours|rate.
Projects
GET /analytics/projects/ Budget burn per project. Returns array: [{project_id, title, client_name, budget_ghs, invoiced_ghs, collected_ghs, total_hours, billable_hours, burn_pct, status}]. burn_pct = (billable_hours * effective_rate) / budget * 100.
Insights (rule-based)
GET /analytics/insights/ Computed rule-based observations about the business. Returns [{type, severity, title, body, entity_type?, entity_id?, value?}]. See insight spec in section 6. These are computed fresh on each request — no storage. Fast enough at Stero's data volume.
Business Assistant (rule-based)
GET /assistant/briefing/ Daily assistant briefing payload for the current organisation. Returns {headline, revenue_summary, follow_up, insights} where follow_up reuses the structured recommendation contract from the assistant router.
POST /assistant/query/ Deterministic business assistant router. Accepts {message, conversation_history?, context?} and returns {reply, matched_rule, data_context_used?, draft?}. Supported Sprint 3 intents: revenue summary, overdue balance summary, proposal summary, client health summary, project risk summary, today's follow-up recommendations, proposal follow-up email draft, overdue invoice reminder draft, and stale project follow-up draft.
Query performance — no optimisation needed yet

At Stero's current data volume (tens of clients, hundreds of invoices), all these queries run in under 50ms with zero extra indexes beyond the FKs you already have. Do not add database indexes, caching, or materialised views in Sprint 3. Profile first, optimise when you can measure a real problem. The only index worth considering now is payment_date on Payment if you find the revenue query slow — but you won't.

Email drafting in Sprint 3 is template-driven. The backend chooses a template based on CRM facts such as proposal status, deadline proximity, overdue duration, and outstanding amount, then fills placeholders with client and project data. This gives you useful drafts without any hallucination risk.

build_ai_context() function

Write this in analytics/services.py. It is not an API endpoint. It is called internally. It becomes the shared context snapshot for Celery jobs, dashboard intelligence callouts, and later rule packs.

def build_ai_context(organisation) -> str:
  # Fetches and serialises a structured snapshot of CRM data.
  # Capped: last 3 months activity, top 10 clients by revenue,
  # open proposals, overdue invoices, active projects with hours.
  # Returns a plain-text summary string, under 2,000 tokens.
  # Never includes payment references or personal contact details.
  # Tested independently from any future consumer.
Hard token cap on the context function

The output of build_ai_context() must stay under 2,000 tokens (roughly 1,500 words). Enforce this with a character count assertion in the test. When you have 500 clients and 5,000 invoices, this function must still return a capped summary — not a full database dump. The cap is not a later-sprint concern; it's a Sprint 3 design constraint you build in from day one. Use recency (last 3 months) and importance (top 10 by revenue) as your two trimming axes.

Proactive insights specification

Five deterministic insight types computed from your CRM data. Each is a pure function: given the data, it produces an observation or says nothing. No fuzzy logic, no LLM. These are the observations you'd make yourself if you reviewed the numbers every morning — automated.

Overdue concentration
Fires when one client accounts for >50% of total outstanding balance. Signal: collection risk is concentrated, not spread.

Revenue trend
Compares this month's collected revenue to the previous 3-month average. Fires when change is >±20%. Positive or negative.

Pipeline stall
Fires when a proposal has been in sent or negotiating status for longer than 2× the average days-to-close. Signal: follow up.
2 proposals
Stalled 34+ days (avg is 18)
Budget burn warning
Fires when a project's billable hours multiplied by effective rate exceeds 80% of the project budget, but the project status is not done. This means you're consuming more time value than was priced. The fix is either to raise the next invoice to cover additional scope, or flag the project for a budget conversation. Threshold: 80% burn. Signal: act before you go over, not after.

Insights return nothing when data is insufficient

Each insight function receives the organisation's data and either returns an insight object or returns None. The revenue trend insight requires at least 4 months of data — if you have less, it stays silent rather than producing a misleading comparison. The pipeline stall insight requires at least 3 decided proposals to establish an average. Always check for sufficient data before computing. An empty insights panel is correct; a wrong insight destroys trust.

Intelligence context layer

The build_ai_context() function is the main shared-infrastructure task in Sprint 3. It is not connected to any model API. Its job in Sprint 3 is to exist, be tested, and be correct so the assistant router, Celery jobs, and later rule-based automations can reuse one trusted summary builder.

What the function must return

A plain-text operating summary. It does not use Markdown headings or JSON — it's a dense paragraph-by-paragraph snapshot that reads clearly for humans, Celery jobs, and rule evaluators. Example output structure:

What Sprint 3 hands to Sprint 4

Sprint 4 is the operations layer. Sprint 3 gives it better numbers, safer alerts, and reusable intelligence plumbing. The list below is what Sprint 4 picks up and builds on.

Sprint 3 delivers to Sprint 4
→build_ai_context(organisation) — tested, correct, capped. Sprint 4 can reuse it inside Celery jobs, contract alerts, expense warnings, and dashboard summaries without rebuilding aggregation logic.
→All analytics endpoints — Sprint 4 can show live financial context inside contracts, expenses, and project operations screens without inventing a second reporting layer.
→Rule-based insights — Sprint 4 can attach them to tasks, reminders, and notification workflows. "Why is GIS margin low?" already has deterministic data behind it.
→Mobile-ready UI — Sprint 4 adds more operational screens, and Sprint 3's mobile foundation prevents those screens from becoming desktop-only regressions.
What Sprint 4 builds on top
+Contracts model + Expenses model — the two new root entities Sprint 4 introduces. Both get organisation FK, AuditLog signal, soft delete — same pattern as everything before them.
+Document attachments, task assignment, and project kanban — all of them can surface the same analytics summaries and insight warnings Sprint 3 already computes.
+Email sending via Celery — a new task reads from the Notification table (built in Sprint 2) and sends emails. Notifications already exist; Sprint 4 adds the delivery mechanism.
+Role-aware operational workflows — multi-user screens can consume the existing rule outputs without changing how intelligence is calculated.
The full four-sprint picture

Sprint 1 built the client and pipeline data. Sprint 2 built the financial data. Sprint 3 aggregates that data into analytics and prepares the intelligence context layer. Sprint 4 expands the operations layer on top of that foundation and reuses the same rule outputs in more workflows. The roadmap stays deterministic: better data first, better rules second, broader automation later.
